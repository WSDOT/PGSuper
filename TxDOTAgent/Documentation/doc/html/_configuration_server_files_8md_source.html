1 Configuration Servers - The Details {#configuration_server_files}
2 ============
3 Configuration Servers are more of a concept than what we traditionally think of as a server. Examples of the servers we traditionally think of are FTP, HTTP, PHP, SQL, and the like. All of these servers use dedicated hardware running specialized software. The Configuration Servers we refer to are basically locations where some files can be stored and accessed by users.
4 
5 Our Configuration Servers can use a local or network file system on your computer, an FTP server or an HTTP server to store and provide access to Configurations.
6 
7 In this topic we will describe how to setup configuration servers.
8 
9 Local and Network File System based Configuration Servers
10 ----------------------------------------------------------
11 Configuration Servers based on a file system are the easiest to setup, however the accessibility is limited to a LAN/WAN. Generally, you do not have your file system shared on the Internet for everyone to use (although it may work on DropBox or Google Drive, we have not tested this). File system based configuration servers are useful when developing configurations or when working with a limited group.
12 
13 To set up the server:
14 1. Determine the location on your file system where you would like the server to reside. Let's say G:\\MyTOGA\\.
15 2. Copy your Master Library to G:\\MyTOGA\\. The Master Library is a file you create with the PGS Library Editor application. The file has a .LBR extension.
16 3. Create a folder for template files, let's say G:\\MyTOGA\\MyTemplates\\
17 3. Copy your TOGA Project Templates (*.togt), icon files (*.ico) and TogaTemplate.pgs to G:\\MyTOGA\\MyTemplates.
18 4. Let everyone know how to access your configuration.
19 
20 > NOTE: Local and Network File System based servers do not use publishing instructions or configuration documentation.
21 
22 See @ref configurations in the @ref adminstrator_guide to learn how to configure TOGA to use a file system based configuration. You will want to provide your users with the following information that is to be entered in the Configuration Server Definition window.
23 
24 Item | Value
25 -----|----------
26 Name | This can be any name. Specify one, or let your users decide
27 Server Type | Local or Network File System
28 Master Library File | The full path to your Master Library file. (e.g. G:\\MyTOGA\\MasterLibrary.lbr)
29 Path to Template Files | The full path to the folder containing your Project Templates. (e.g. G:\\MyTOGA\\MyTemplates\\). The TOGA Project Templates folder contains all of your templates (*.togt), icon files (*.ico) and TogaTemplate.pgs
30 
31 FTP and HTTP based Configuration Servers
32 ----------------------------------------
33 We have a little more work to do before setting up an FTP or HTTP based configuration server.
34 
35 ### Configuration Documentation ###
36 Let's start by writing the optional configuration documentation. This is simply an HTML page that contains information about your configuration. You can create any web page you want, but the documentation generally should contain the following information:
37 * Description
38 * Publishing Date
39 * Warnings and Limitations
40 * Support contact information
41 * Revision log
42 
43 The configuration documentation must be placed on an HTTP or FTP server where users can access it. The location of the configuration documentation is recorded in the publishing instructions as described below. The @ref dialogs_configure window has a hyperlink that, when pressed, will display your configuration documentation.
44 
45 ![](ConfigureWindow.png)
46 
47 ### Building the Configuration Package ###
48 The Master Library and Project Templates must be placed into a Configuration Package file (*.pgz) that will be stored on the configuration server. The configuration package allows the software to download a single file when installing a configuration, rather than downloading the Master Library file and additional files for each Project Template. TOGA validates the file contents using an mD5 checksum to insure that the file contents have not been corrupted during download.
49 
50 The MakePGZ.exe utility program was installed with BrigeLink. It can be found in the BridgeLink application folder. MakePGZ creates the Configuration Package.
51 
52 To make a configuration package, use the command:
53 
54  MakePGZ.exe pgz_file master_library_file project_template_folder TOGT PGS
55 
56 where
57 
58 Parameter | Description
59 ----------|------------
60 pgz_file | The name of the TOGA Configuration Package (PGZ file) that will be created
61 master_library_file | The name of your Master Library file
62 project_template_folder | The name of the folder containing your project templates (*.togt files), icon files (*.ico) and TogaTemplate.pgs.
63 TOGT PGS | The file extension for your templates and pgs file (use TOGT PGS literally)
64 
65 Example:
66 
67  C:\Program Files\WSDOT\BridgeLink\MakePGZ.exe C:\MyFolder\MyState.pgz G:\MyTOGA\MasterLibrary.lbr G:\MyTOGA\MyTemplates\ TOGT PGS
68 
69 This command will generate two files: C:\\MyFolder\\MyState.pgz, and C:\\MyFolder\\MyState.pgz.mD5. Both of these files should be published onto your server.
70 
71 > Note: The template folder is a single folder containing all of your *.togt files, *.ico files, and TogaTemplate.pgs. This is unlike how a multi-folder structure is used in PGSuper or PGSplice templates. Also, note that MakePGZ.exe is dumb: it does not verify that all of the files necessary to run TOGA are present.
72 
73 ### Publishing Instructions ###
74 Next, we have to create the publishing instructions file. Publishing instructions are provided in a specially-named text file called TOGAPackages.ini. The content of the file provides information about the configurations you are publishing.
75 
76 Here is an example TOGAPackages.ini file for an FTP server:
77 
78  [WSDOT]
79  WebLink="http://www.wsdot.wa.gov/eesc/bridge/software/TOGA/LibraryInfo.html"
80  Format="pgz"
81  Version_2.9.1_PgzFiles="ftp://ftp.wsdot.wa.gov/public/bridge/Software/TOGA/Version_2.9.1/WSDOT.pgz"
82  Version_3.0.0_PgzFiles="ftp://ftp.wsdot.wa.gov/public/bridge/Software/TOGA/Version_3.0.0/WSDOT.pgz"
83 
84  [AASHTO]
85  WebLink="http://www.wsdot.wa.gov/eesc/bridge/software/TOGA/LibraryInfo.html"
86  Format="pgz"
87  Version_2.9.1_PgzFiles="ftp://ftp.wsdot.wa.gov/public/bridge/Software/TOGA/Version_2.9.1/AASHTO.pgz"
88  Version_3.0.0_PgzFiles="ftp://ftp.wsdot.wa.gov/public/bridge/Software/TOGA/Version_3.0.0/AASHTO.pgz"
89 
90 Here is an example TOGAPackages.ini file for a file system server:
91 
92  [TxDOT]
93  Format="pgz"
94  WebLink="http://ftp.dot.state.tx.us/pub/txdot-info/brg/TOGA/TxDOT_TOGA_LibraryInfo.htm"
95  Version_2.9.1_PgzFiles="C:\Shared\TogaServers\txdot.pgz"
96  Version_3.0.0_PgzFiles="C:\Shared\TogaServers\txdot30.pgz"
97 
98 Instructions for a configuration begin with the configuration name in square brackets. For the FTP example, publishing instructions are for two configurations; WSDOT and AASHTO.
99 
100 Following the configuration name are the publishing instructions for the configuration. The instructions are a keyword-value pair.
101 
102 Keyword | Description
103 --------|-----------
104 WebLink | URL of the configuration documentation. Omit if configuration documentation is not used.
105 Format | Format of the configuration package. "pgz" is the only valid value at this time
106 Version_x.y.z_PgzFiles | Location of the configuration package. Substitute the application version number for x.y.z. In the examples above, configuration packages are available for TOGA version 2.9.1 and 3.0.0
107 
108 When installing a configuration, TOGA will look for the configuration package with the highest version number that is not greater than its own version number. If you are use TOGA Version 2.9.5, the configuration package for version 2.9.1 will be installed.
109 
110 > NOTE: All configuration packages in an .ini file must be of the same type (e.g., links in an .ini file on an FTP server must all be FTP links). The server type must match the type defined in the @ref configuration_server_definition.
111 
112 Use NOTEPAD.EXE or any other plain text editor to create your publishing instructions. Be sure to save your publishing instructions with the name TOGAPackages.ini
113 
114 ### Publishing your Configuration ###
115 Your configuration is now ready to be published on the configuration server and accessed by engineers all around the world.
116 
117 To publish your configuration:
118 1. Open Windows Explorer or an ftp client to the location on your HTTP server where the configuration documentation will be stored. Copy the documentation onto the server
119 2. Open Windows Explorer or an ftp client to the location on the server where you will be storing the publishing instructions and configuration packages. Copy the publishing instructions (TOGAPackages.ini) and the configuration packages (the .PGZ and .PGZ.mD5 files).
120 3. Let everyone know how to access your configuration.
121 
122 See @ref configurations to learn how to configure TOGA to use an FTP or HTTP based configuration. You will want to provide your users with the following information that is to be entered in the PGSuper Configuration Server Definition window.
123 
124 Item | Value
125 -----|----------
126 Name | This can be any name. Specify one or let your users decide
127 Server Type | The server type you are using, either Internet FTP Server, Internet HTTP Server, or file system.
128 Server Address (URL) | The fully qualified URL for your server. This is the location where the INI file is stored. (e.g. http://www.mycompany.com/PGSuper/ConfigurationServer/)
